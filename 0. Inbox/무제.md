## 작업 범위: 기록 페이지 (History View)
- HistoryView는 Game 모듈의 View다(Bedrock.UI의 View 베이스 상속). 2단계 MainView와 같은 방식.
- 1단계에서 만든 기록 저장/로드(Data)를 읽어 과거 추첨 목록을 표시한다.
- UI Toolkit만(UXML + USS + 코드비하인드), Canvas 금지. CLAUDE.md/ARCHITECTURE.md 준수.

### 1) HistoryView 레이아웃 (UXML + USS)
- 제목 / 뒤로가기 버튼 / 과거 추첨 리스트 / 빈 상태 메시지
- 각 항목: 회차, 예상번호 6개 + 보너스, 생성 날짜, 판정 상태 배지
- 번호 타일은 2단계 MainView의 색상대 스타일을 재사용(복붙 말고 공유 가능하면 공유)
- 리스트는 UI Toolkit ListView(가상화)로 구성 + 항목 UXML 템플릿 + 바인딩
- 최신순 정렬

### 2) 데이터 연동
- View가 열릴 때(OnShow)에 1단계 기록 서비스에서 최신 목록을 로드해 렌더
- 추첨 후 다시 들어오면 목록이 갱신되도록 매번 fresh 로드

### 3) 판정 상태 표시
- 상태 배지가 모든 경우(대기 / 불일치 / 일치+등수)를 표현하도록 지금 만들어 둘 것
- 단, 실제 판정 데이터는 5단계(네트워크) 전까지 대부분 "대기"로 표시됨

### 4) 내비게이션
- 2단계에서 자리만 둔 MainView의 기록 버튼 → UIManager로 HistoryView를 push
- 뒤로가기 버튼 / 안드로이드 back → pop으로 MainView 복귀
- 화면이 2개일 때 View 스택이 제대로 동작하는지 확인

### 5) (선택) 전체 삭제
- 1단계에 전체 삭제 API가 있으면 "기록 비우기" 버튼 + 확인 팝업(공통 다이얼로그) 연결

### 검증 / 발견 포인트
- 두 번째 Game View + 내비게이션(push/pop/back)이 매끄러운지
- UI Toolkit ListView가 View 프레임워크와 깔끔히 붙는지
- 번호 타일을 MainView와 HistoryView에서 재사용할 수 있는지
  (복붙해야 한다면 UI 컴포넌트 재사용 측면의 템플릿 보완점으로 메모)
- OnShow에서 데이터 갱신이 잘 되는지
- 막히거나 어색한 부분 메모
